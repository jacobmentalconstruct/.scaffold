# `config/toolbox_manifest.json` — PROSE PLAN

> **Status:** Tranche B plan. The actual JSON is generated at install time and refreshed on every tool/package register/unregister. This file describes the planned shape and discipline.

## Purpose

A **single high-level descriptor of the entire `.scaffold/` toolbox**, designed for **zero-context agent entry.** When an agent connects to a sidecar it has never seen before, it should read `contracts/BCC.md` first, then the local binding artifact, and then this manifest. From it, the agent learns:

- What this toolbox is and what it's for.
- What tools are available and at what authority levels.
- What projections it can read.
- What the contract requires of an agent before doing meaningful work.
- Where to find everything else.

Adapted from the precursor's `toolbox_manifest.json` pattern (see `_docs/migration/INCORPORATION_INVENTORY.md` §5.3).

## Generation discipline

- Generated by `install_orchestrator` on first boot and refreshed by `tool_registry_manager` whenever a tool registers/unregisters or a contract revision is acknowledged.
- The file is **derived state** (per [`cache/README.md`](cache/README.md) discipline): if deleted, it can be regenerated from the DB. It lives in `config/` (not `cache/`) only because human inspection / external tooling expects it there.
- The `generated_at` timestamp is the truth-tag; readers should compare against `journal_meta.last_change` to know if it's stale.

## Consumers

- **MCP clients** (agents) — the first generated manifest read after `BCC.md` and the local binding artifact.
- **Human readers** — quick "what is this thing?" reference.
- **External tooling** — anything that wants to introspect a `.scaffold/` directory programmatically.

## Non-goals

- Does not contain operational config (that's `sidecar.json`).
- Does not contain per-tool details (that's `tool_manifest.json`).
- Does not replace the repo-local binding artifact — that local concrete data belongs in `_docs/reference/PROJECT_BINDINGS.md`.
